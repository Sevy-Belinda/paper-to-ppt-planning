# Paper To PPT Planning

## 概述

这个 skill 用来把论文转成“真正适合讲”的 PPT 方案。

它的默认职责是策划与结构设计，不是直接承担所有文件制作步骤。

它会先判断：

- 这场汇报是做什么的
- 听众是谁
- 该用什么叙事路线
- 每一页应该承担什么讲述任务
- 当前环境是否具备读取 PDF 和生成 PPTX 的能力

如果环境齐全，它可以作为总控入口触发完整流水线；如果环境不齐全，它会暂停自动化流程并给出下载或降级方案。

## 何时使用

当用户有以下需求时使用：

- 把论文整理成 PPT 内容、汇报大纲或讲稿
- 把现有的论文 Markdown 稿件改成更适合汇报的结构
- 在问题驱动、技术演化、结果导向等叙事路线之间做选择
- 针对课堂、组会、答辩、面试、自学等不同场景重写同一篇论文
- 希望从 PDF 出发自动完成“读取 -> 策划 -> 生成 PPTX”流程
- 决定背景铺垫、公式保留、实验讲法和图片使用方式

以下情况不应优先使用本 skill：

- 只是简单总结论文
- 结构已经定好，只需要直接编辑 `.pptx`
- 只做视觉美化

## 核心规则

**不要一上来就写幻灯片内容，先把整场汇报的设计定下来。**

至少先明确这 5 件事：

1. 汇报目标
2. 听众层级
3. 叙事路线
4. 页数范围
5. 公式密度

如果其中有任何一项不清楚，就先问一个高影响问题，不要直接写完整大纲。

## 环境与工具检查机制

在开始执行任务前，先检查当前工作区是否安装了或可调用处理 PDF 和 PPTX 的相关 skill 或工具。

优先检查三类能力：

- 读取 PDF 的能力，例如 `pdf` skill
- 生成或编辑 PPTX 的能力，例如 `pptx` skill
- 通用文档处理与补充资料能力，例如 `docs` skill

### 推荐下载来源

当本机缺少通用能力时，优先提示用户从以下仓库获取：

- `https://github.com/anthropics/skills`
  推荐使用其中的：`pdf`、`docs`、`pptx`

当任务涉及专业科研领域、需要更强的论文场景支持时，可额外提示用户使用：

- `https://github.com/K-Dense-AI/scientific-agent-skills`

### 如果工具齐全

- 直接进入全自动流水线
- 默认顺序是：`读取 PDF -> 规划大纲 -> 生成 PPTX`

### 如果缺少工具

- 先暂停自动流水线
- 明确告知用户检测到本地缺少 `pdf`、`docs`、`pptx` 中的一项或多项
- 礼貌询问用户是否需要下载

推荐提示语：

- “检测到本地没有可用的 `pdf`、`docs` 或 `pptx` skill。是否需要我先帮助您下载相关能力，再继续完成从 PDF 到 PPT 的流程？”

如果用户同意下载：

- 优先使用 `anthropics/skills` 下载 `pdf`、`docs`、`pptx`
- 如果任务明显偏科研专业场景，可补充推荐 `K-Dense-AI/scientific-agent-skills`
- 下载或安装完成后，继续用户原始需求

如果用户不想下载：

- 提示用户可以直接提供论文文本
- 推荐提示语：
  “您可以直接把 PDF 的文本内容粘贴给我，我可以先帮您生成适合制作 PPT 的 Markdown 结构。”

## 输入处理

可以接受这些输入：

- 论文 PDF
- 论文链接
- 摘要或提取后的正文
- 现有 Markdown、讲稿或笔记
- 用户自己设想的讲法

如果用户提供的是 PDF，且本机存在 `pdf` skill：

- 先调用 `pdf` 提取你需要的论文内容
- 再回到本 skill 完成策划

如果用户只提供文本、Markdown 或笔记：

- 直接基于这些内容工作

如果用户最终要生成 `.pptx`，且本机存在 `pptx` skill：

- 先完成策划输出
- 再把结果交给 `pptx`

如果 `pdf`、`docs` 或 `pptx` 不可用：

- 进入降级模式继续工作
- 读不了论文时，要求用户补充最少必要文本
- 做不了演示文件时，输出完整 Markdown 方案供后续制作

如果用户明确要求“从 PDF 直接生成 PPTX”，但本机缺少相应能力：

- 先触发上面的环境与工具检查机制
- 不要假装已经具备端到端自动化能力

## 工作流

### 第一步：识别任务类型

先判断当前属于哪一类：

- 从零开始的论文转 PPT 策划
- 重写已有 Markdown 讲稿
- 在多条叙事路线之间比较
- 压缩或扩展现有结构
- 全自动流水线前的环境检查与分流

用一句简短的话说明你的判断，让用户知道你当前的工作框架。

### 第二步：锁定策划上下文

一次只问一个问题。推荐按这个顺序收敛：

1. 这是事实校正、结构重写，还是两者都要
2. 汇报场景是什么
3. 听众基础如何
4. 预计多少页
5. 是否已有偏好的叙事路线
6. 公式保留到什么程度
7. 是否需要配图建议

除非用户明确要求快速收集完整条件，否则不要一次抛出所有问题。

### 第三步：给出 2-3 条叙事路线

在正式写结构前，先给出路线与取舍。

常见路线包括：

- 问题驱动
- 技术演化
- 结果导向

每条路线都要解释：

- 它强调什么
- 适合哪类听众
- 它的风险是什么

最后给出推荐方案。

### 第四步：搭建整场汇报的页级逻辑

把整场汇报设计成一条叙事链，而不是事实堆叠。

每一页都应该有清晰任务，例如：

- 建立问题
- 解释旧方法
- 引出转折点
- 解释新方法
- 用实验验证
- 总结影响和结论

如果某页说不清它的职责，就先修结构，不要急着写正文。

### 第五步：输出结构化结果

默认输出 Markdown，因为它最适合 review、修改和后续转换。

页面结构模板在 `references/page-structure-template.md`。

默认每页字段：

- `页标题`
- `标题`
- `页面正文`
- `配图建议`
- `图片来源`
- `讲解提示`

需要始终区分两类内容：

- `页面正文`：面向最终观众
- `配图建议` / `图片来源` / `讲解提示`：面向制作和讲述过程

如果后续真的要做成 PPT，默认会上页的通常只有：

- `页标题`
- `标题`
- `页面正文`
- `配图`

不要默认把 `讲解提示` 或 `图片来源` 塞进幻灯片正文。

### 第六步：做一致性检查

在认为方案完成之前，检查：

- 路线是否与听众和场景匹配
- 背景铺垫是否超出必要程度
- 公式密度是否符合确认结果
- 实验页是在证明论文价值，而不是只报数字
- 输出是否正确区分了上页内容与制作备注

## 路线选择指引

当路线仍然不明确时，读取 `references/route-patterns.md`。

默认判断规则：

- 当听众更关心论文动机、价值和证据时，用 `problem-driven`
- 当听众必须理解新方法如何替代旧方法时，用 `technical-evolution`
- 当时间很紧、必须突出结果和结论时，用 `result-driven`

## 领域补充指引

如果论文属于机器学习、深度学习或相邻 AI 方法，读取 `references/ml-paper-guidelines.md`。

重点纪律：

- 背景是为主方法服务的，不是拿来单独开课的
- 旧方法只讲到足以理解新方法为止
- 实验页要回答“为什么应该相信这篇论文”

## 输出模式

根据用户需求选择最轻且足够的输出方式。

### 讨论模式

适用于用户还在讨论方向。

输出内容：

- 当前理解的简要总结
- 2-3 条路线选项
- 你的推荐
- 下一个关键问题

### 策划模式

适用于用户要完整 PPT 结构，但暂时不做实际幻灯片。

输出内容：

- 逐页 Markdown 方案
- 必要时补充配图建议
- 必要时补充讲解提示

### 重写模式

适用于用户已经有一版 Markdown 讲稿。

输出内容：

- 哪些部分应该保留
- 哪些部分应该合并、拆分、移动或重写
- 更新后的页级结构

## 常见错误

### 错误：把总结当成汇报

表现：

- 每一页都只是知识点堆积

修正：

- 重新定义每页的叙事任务

### 错误：背景部分失控

表现：

- 前半场变成旧方法教程

修正：

- 只保留理解论文贡献所必需的背景

### 错误：没有做听众校准

表现：

- 要么术语过多，要么过度简化

修正：

- 回到听众层级，重新控制深度

### 错误：实验页只是报分

表现：

- 只列指标，不解释这些实验在证明什么

修正：

- 围绕“实验支撑了什么结论”重写页面

### 错误：制作备注泄漏到幻灯片

表现：

- 图片来源、讲者提示混进上页正文

修正：

- 严格分离观众内容与制作内容

## 降级规则

如果无法完整读取论文：

- 明确说明缺少什么输入
- 只请求最小必要补充
- 如果条件允许，仍继续完成路线设计和结构搭建

如果用户要最终 `.pptx`，但本机没有 `pptx` skill：

- 仍然输出完整的 Markdown 页级方案
- 让它足够干净，便于手工或后续自动转换

## 示例锚点

`Attention Is All You Need` 是这个 skill 的强示例，因为它天然支持多条有效路线：

- 技术演化线：`Seq2Seq -> RNN -> Attention -> Transformer -> Experiments -> Impact`
- 问题驱动线：`What problem the paper solves -> Why old methods struggle -> Why Transformer works -> Evidence -> Influence`

把这类模式当作锚点，而不是把 workflow 写死在这篇论文上。
